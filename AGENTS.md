# AGENTS.md

本文件面向在本仓库内工作的 AI / 自动化代理。目标是：在**不误判策略语义、不破坏回测结果、不泄露敏感信息、不绕过部署流程**的前提下，安全地开展开发、调试、测试、文档更新与部署协作。

---

## 1. 作用范围与最高优先级

本文件适用于仓库根目录及其所有子目录，尤其是：

- `backtest.py`
- `scan_next_b.py`
- `evaluate_ratchet.py`
- `sweep_ratchet.py`
- `web_app.py`
- `web_site/`

代理在本仓库内工作时，优先遵守以下原则：

1. **先保护策略语义，再做功能改动**
   - 不要在未获明确批准时改变信号判定、交易时点、仓位语义、费用模型、报告口径。
   - 本项目最核心的执行假设之一是：**信号出现后的下一交易日开盘价执行交易**。如需调整，必须先征得用户明确同意。

2. **先保护本地可用性，再考虑部署扩展**
   - `web_app.py` 是当前本地业务页入口。
   - `web_site/` 是新增的 Web 登录壳与前置代理层。
   - 不要为了 Web 部署便利而破坏本地使用流程。

3. **先走 Git 流程，再做服务器动作**
   - 允许的标准流程：**本机修改 -> 本机测试 -> 本机 git commit -> 本机 git push -> 服务器 git pull -> 按明确命令部署/重启**。
   - 禁止在服务器端直接改源码、手工覆盖文件、临时热修代码、偷偷改配置或数据。

4. **先小改、可验证，再逐步扩大**
   - 优先做小步、可回滚、可测试的修改。
   - 修改后必须给出实际验证结果，而不是只说“理论上应该可以”。

## 2. 项目概览

这是一个围绕 **MA5 / 主升浪交易逻辑**构建的本地回测与信号扫描项目，当前主要包含两类能力：

1. **回测**：读取本地 CSV，或通过 `akshare` / `yfinance` 拉取行情，生成交易明细、权益曲线和 HTML 报告。
2. **扫描**：扫描“下一交易日可执行的 B 点候选”。

当前已知的策略版本包括：

- `classic`（经典版）
- `ratchet`（棘轮版）

当前 Web 架构分为两层：

- `web_app.py`：原有本地业务 Web 页面，默认监听 `127.0.0.1:8765`
- `web_site/`：新增登录壳与前置代理，默认监听 `127.0.0.1:8764`

推荐的部署拓扑为：

`公网域名 / 反向代理 -> web_site -> 仅本机可访问的 web_app.py`

**不要让公网直接访问 `web_app.py`。**

## 3. 仓库结构速览

### 核心脚本

- `backtest.py`
  - 核心回测入口。
  - 负责读取数据、生成信号、执行回测、汇总结果、输出 `trades.csv` / `equity.csv` / `report.html`。

- `scan_next_b.py`
  - 扫描最新日 K 是否出现 B 点，以及下一交易日可执行的候选标的。

- `evaluate_ratchet.py`
  - 用于比较 `classic` 与 `ratchet` 两种策略版本的结果。

- `sweep_ratchet.py`
  - 用于扫描棘轮参数组合。

### 本地 Web 业务层

- `web_app.py`
  - 原有本地页面入口。
  - 提供回测表单、扫描表单、候选详情等页面。
  - 这是当前业务展示层的主入口，不要轻易改动页面语义或路由行为。

### Web 登录壳 / 代理层

- `web_site/app.py`
  - 登录入口、会话校验、错误页、代理转发。

- `web_site/auth.py`
  - 密码校验与内存会话管理。

- `web_site/proxy.py`
  - 把已认证请求转发给 `web_app.py`。

- `web_site/constants.py`
  - 端口、Cookie、登录路径、超时、默认密码、登录页样式等常量。

- `web_site/static/login.html`
  - 登录页模板。

### 文档与辅助文件

- `README.md`
  - 面向使用者的项目说明与 CLI 用法。

- `web_site/README.md`
  - 面向 Web 登录壳的本地运行与部署说明。

- `sample.csv`
  - 离线烟雾测试用样例数据。

- `start_strategy_tester.bat`
  - 本地启动辅助脚本。
  - 注意：此类脚本可能包含机器相关路径，修改时必须核对当前环境，不要盲目信任硬编码解释器路径。

### 生成物 / 临时文件

以下内容通常视为**生成物**或**临时文件**：

- `_tmp_trades.csv`
- `_tmp_equity.csv`
- `_tmp_report.html`
- `__pycache__/`
- `*.zip`

除非用户明确要求，否则不要把这些文件当作业务源码来改。

## 4. 沟通、文案与命名规范

1. **默认使用中文沟通**
   - 面向用户的解释、分析、风险提示、方案说明、提交说明，默认使用中文。
   - 代码、文件名、类名、函数名、参数名、库名、命令本身等，可保留必要英文。

2. **讨论策略字段时，必须附中文释义**
   - 推荐格式：`field_name`（中文说明）。
   - 例如：
     - `ma_length`（生命线周期）
     - `vol_length`（平均成交量周期）
     - `vol_multiplier`（巨量倍数）
     - `stop_5ma_pct`（跌破 5 日线容忍百分比）
     - `hard_stop_pct`（B 点追踪止损百分比）
     - `reentry_pct`（反抽重返距离百分比）
     - `commission_pct`（单边手续费百分比）
     - `slippage_pct`（单边滑点百分比）

3. **Web 页面文案默认中文，品牌标题可保留英文**
   - 例如 `MA5 Strategy Hub` 可保留英文。
   - 其余按钮、说明、错误提示、状态文案默认优先中文。

4. **用户可见页面不要暴露内部实现细节**
   - 不要在登录页、错误页或产品化界面中直接写出内部文件名、模块名、临时实现方式、默认密码来源等。
   - 用户界面应优先展示业务层面的提示，而不是开发者内部细节。

5. **Git 提交日志必须使用中文**
   - commit message 必须准确、简洁地描述改动目的。
   - 避免使用 `fix bug`、`update stuff` 之类含糊英文短句。

6. **不要在 `AGENTS.md` 中记录具体收益数字**
   - 描述策略时，只写结构、约束、参数含义、开发规则。
   - 不要在本文件中记录具体收益率、夏普、回撤等会快速过期的数值。

## 5. 允许改动与禁止改动

### 可以在不额外请示的前提下进行的改动

- 文档补充、结构梳理、注释优化
- 不改变行为的重构
- 明显的 bug 修复
- `web_site/` 下的 UI 美化、文案收敛、前置登录壳体验优化
- Linux / Windows 跨平台兼容修复（前提是不改变策略逻辑和页面语义）
- 小范围测试代码或验证脚本（完成后应清理或说明）

### 必须先得到用户明确批准的改动

- 修改信号判定逻辑
- 修改 `classic` / `ratchet` 的默认参数
- 修改交易执行时点
- 修改初始资金、手续费、滑点、仓位模型等影响回测结果的默认设定
- 修改报告统计口径或图表语义
- 修改扫描器筛选门槛或默认股票池语义
- 让 `web_app.py` 直接暴露到公网
- 将密码、会话、认证方式改成持久化存储或外部服务依赖
- 直接改服务器上的代码、配置、数据库、缓存或运行态文件

### 对 `web_app.py` 的特殊约束

- `web_app.py` 是当前本地业务页主入口。
- 不要无批准地改其核心路由、页面语义、表单含义、策略调用方式。
- 允许做**窄范围跨平台兼容修复**，但必须满足：
  1. 不改变策略逻辑；
  2. 不改变页面业务语义；
  3. 不影响 Windows 本地使用。

### 对 `web_site/` 的特殊约束

- 所有新增的 Web 认证、登录壳、反向代理相关逻辑，优先放在 `web_site/` 下。
- 不要把原有策略逻辑复制一份放到 `web_site/` 里重写。
- `web_site/` 的职责是**访问控制与流量转发**，不是再造一套策略系统。

## 6. 代码与实现约定

1. **优先使用标准库或跨平台实现**
   - 若同一需求可通过 Python 标准库完成，不要优先依赖 PowerShell 专用实现。
   - 目标是同时兼容 Windows 与 Linux。

2. **不要为临时方便而引入强平台绑定方案**
   - 如确需依赖平台专属能力，必须在文档里说明前提条件和替代路径。

3. **新增依赖前要判断是否真的必要**
   - 能不用就不用；能轻量就不要厚重。
   - 如果新增依赖影响运行方式或部署方式，必须同步更新文档。

4. **避免无关格式化和大面积重排**
   - 修改应聚焦任务本身。
   - 不要顺手重排整个文件、改无关命名、批量变动不必要的缩进或空行。

5. **保留已验证的业务分层**
   - 回测 / 扫描逻辑在核心脚本中。
   - 本地业务界面在 `web_app.py`。
   - 对外登录入口与认证代理在 `web_site/`。

## 7. 常用运行与验证方式

### 离线回测烟雾测试（优先）

优先使用 `sample.csv` 做离线验证，避免把网络问题误判为代码问题：

```text
python backtest.py --csv sample.csv --trades-out _tmp_trades.csv --equity-out _tmp_equity.csv --report-out _tmp_report.html
```

适用场景：

- 修改 `backtest.py`
- 调整报告生成逻辑
- 验证输出文件结构未被破坏

### Web 本地联调

如果改动涉及 `web_app.py` 或 `web_site/`，推荐至少完成以下链路验证：

1. 启动 `web_app.py`
2. 启动 `python web_site/app.py`
3. 打开 `http://127.0.0.1:8764/login`
4. 验证登录页可访问
5. 验证登录成功后能进入业务页
6. 如改动涉及登出、错误页、会话逻辑，则一并验证对应分支

### Python 语法验证

修改 Python 文件后，至少应执行相关文件的编译检查，例如：

```text
python -m py_compile backtest.py scan_next_b.py evaluate_ratchet.py sweep_ratchet.py web_app.py web_site/app.py web_site/auth.py web_site/proxy.py web_site/constants.py
```

如果只改了部分文件，也可以只检查相关文件，但**不要跳过最基本的语法验证**。

### 网络依赖提醒

以下场景可能依赖外部服务：

- `akshare` 拉数
- `yfinance` 拉数
- Nasdaq 股票池相关抓取

如果相关测试失败，先判断是外部接口问题、网络问题还是代码回归，不要直接把外部波动误判为本地逻辑错误。

文档更新的目标是：**让后来者只看文档，也能理解当前真实状态**。

## 9. 部署与服务器规则

1. **服务器部署必须走 Git 流程**
   - 正确顺序：本机修改、本机测试、本机提交、本机推送、服务器拉取、按明确命令部署。

2. **禁止无批准的服务器端修改**
   - 禁止 SSH 上线后直接改源码。
   - 禁止手工上传覆盖文件。
   - 禁止随意修改数据库、缓存、配置、运行目录数据。

3. **如需改服务器状态，必须同时满足两个条件**
   1. 有明确、可审计的命令；
   2. 已获得用户明确批准。

4. **Web 线上部署保持双层隔离**
   - 对外域名应先进入站点栈 / 反向代理，再进入 `web_site`，最后由 `web_site` 代理到仅本机可访问的 `web_app.py`。
   - 不要把 `web_app.py` 直接绑定到公网地址。

5. **如涉及 Git 远程拉取，优先使用 SSH 方案**
   - 若服务器端已配置 Deploy Key，则优先保持 SSH remote。
   - 不要依赖需要人工交互输入凭据的 HTTPS pull 流程。

6. **不要把敏感信息写入仓库**
   - 包括但不限于：私钥、口令、Session、Cookie、服务器专有配置、真实凭据。

## 10. 需要先停下来问用户的场景

出现以下情况时，不要自己拍板，先问用户：

- 不确定某个改动是否已经改变策略语义
- 需要修改默认参数，尤其是会影响回测结果的参数
- 需要新增较重依赖或改变部署方式
- 需要修改登录模型、认证方式、会话持久化方式
- 需要将本地工具改造成公网长期运行服务
- 需要调整域名、端口暴露方式、反向代理结构
- 需要直接操作服务器上的文件、服务、配置或数据
- 发现现有文档与代码明显冲突，且无法确认哪一方是最新事实

原则：**凡是可能改变结果解释、部署安全、使用习惯或服务器状态的事项，先确认。**

## 11. 代理工作建议流程

推荐按下面顺序工作：

1. 先阅读 `AGENTS.md`、`README.md`、相关源码
2. 明确本次任务是否触碰策略逻辑、部署边界或公网暴露问题
3. 小步修改，尽量只改与任务直接相关的文件
4. 先做语法验证，再做最小可行链路验证
5. 若修改影响说明文档，同步更新文档
6. 总结时明确写出：改了什么、没改什么、验证了什么、还有什么风险

如果一个改动会同时影响“策略逻辑 + Web 页面 + 部署方式”，通常说明范围已经过大，应拆分任务，分步推进。
